# STEP 1 — Observation (관찰)

워크스페이스 `MagicSquare_`는 아직 비어 있습니다. 즉, **코드보다 “무엇을 위해 이걸 하는가”를 먼저 정의하는 단계**에 있습니다. 아래는 “4x4 마방진 프로그램을 만든다”는 표현 없이, **관찰 관점**으로 서술한 답입니다.

---

## 1. 현재 우리가 해결하려는 상황은 무엇인가?

**관찰된 상황:**

- 한 주니어 개발자가 **실제로 돌아가는 소프트웨어를 만들며 역량을 쌓고 싶어** 하지만, 연습용 토이 프로젝트와 “사람이 쓸 만한 것” 사이에서 기준이 흐릿합니다.
- **Cursor AI 같은 도구를 “코드 생성기”가 아니라 협업 도구로 쓰는 법**을 익히려는 시도가 있습니다. (요구 정의 → 구현 → 검증의 흐름을 AI와 함께 반복하는 경험)
- **수학·퍼즐·알고리즘 문제**는 “정답이 명확한” 영역이라, 학습과 검증이 쉽습니다. 반면 **누가, 언제, 왜 이걸 쓰는지**는 아직 드러나지 않았습니다.
- 프로젝트 폴더 이름(`MagicSquare_`)은 **4×4 격자 + 행·열·대각선 합 일치** 같은 **제약 조건을 만족하는 배치** 문제에 관심이 있다는 신호일 뿐, **구체적인 사용자·상황·불편**은 아직 관찰되지 않았습니다.

**한 줄로:**  
“개발 실력 + AI 활용 + 검증 가능한 도메인”을 동시에 만족시키려는 **학습·실험 상황**이며, **아직 ‘누구의 어떤 문제’인지는 관찰되지 않았습니다.**

---

## 2. 왜 4×4 마방진 문제를 다루는가?

**“마방진을 만든다”가 아니라, 왜 이 **유형의 문제**가 선택되었는지:**

| 관찰 | 의미 |
|------|------|
| **규모가 작다 (4×4, 숫자 1~16)** | 상태 공간·UI·테스트 범위가 주니어에게 다루기 쉬움 |
| **규칙이 명확하다** | “맞다/틀리다”를 프로그램으로 자동 검증하기 좋음 |
| **알고리즘이 있다 (예: Siamese, Dürer 등)** | “생성”과 “검증”을 분리해 설계·테스트 연습 가능 |
| **시각적으로 직관적** | 격자 UI, 하이라이트, 합 표시 등 **사용자에게 보여줄 단서**가 풍부 |
| **교육·퍼즐 맥락과 잘 맞음** | 학생, 교사, 퍼즐 애호가 등 **잠재 사용자**를 상상하기 쉬움 |

**주의 (The Mom Test 관점):**  
“4×4가 적당해서”는 **개발자 편의**에 가깝습니다. **사용자가 4×4를 꼭 필요로 하는지**는 아직 관찰·검증되지 않았습니다. 3×3, 5×5, 또는 “마방진이 아닌 다른 제약 퍼즐”이 더 맞을 수도 있습니다.

---

## 3. 이 문제는 어떤 학습 또는 시스템 설계 맥락에서 등장하는가?

### 학습 맥락

- **초·중등 수학:** 대칭, 합, 패턴, 논리적 추론
- **알고리즘·자료구조:** 백트래킹, 제약 만족, brute force vs 휴리스틱, **불변 조건(invariant)** 검증
- **소프트웨어 공학 연습:** 입력 검증, 상태 표현(2D 배열), 순수 함수 vs UI, 단위 테스트 vs 통합 테스트
- **AI 협업 연습:** “문제 정의 → 스펙 → 구현 → 리팩터”를 Cursor와 단계별로 나누는 **프로세스 학습**

### 시스템 설계 맥락 (나중에 설계할 때 자연스럽게 등장하는 질문들)

- **생성 vs 검증 vs 힌트 vs 풀이** — 역할을 나눌지, 한 앱에 몰지
- **상태 모델** — 격자 한 칸, 한 행, 전체 보드 중 무엇이 “진실의 원천”인지
- **피드백 루프** — 틀렸을 때 “왜 틀렸는지”를 어떻까지 설명할지 (합만? 어느 줄? 가능한 수정?)
- **난이도·모드** — 빈 칸 채우기, 드래그 배치, 시간 제한, 단계별 힌트 등
- **확장성** — 4×4만 고정할지, n×n으로 일반화할지 (학습 목표와 트레이드오프)

이 맥락에서 **“4×4 마방진”은 도메인 객체**일 뿐이고, 실제로 다루게 되는 것은 **제약을 만족하는 배치를 다루는 작은 결정/피드백 시스템**입니다.

---

## 관찰 요약 (Mom Test 스타일)

지금까지 **관찰 가능한 사실**만 모으면:

1. **당신의 상황:** 주니어 개발자로서 AI와 함께 **작고 검증 가능한** 프로젝트로 실력을 키우려 함.
2. **선택된 문제 유형:** 작은 격자 + 명확한 규칙 + 시각화·알고리즘·교육과 연결 가능.
3. **아직 관찰되지 않은 것:** 특정 사용자가 **어느 순간** 무엇 때문에 **막히거나 시간을 쓰는지** (교실? 혼자 공부? 퍼즐 앱? 면접 준비?).

# STEP 2 — Why #1

## 가정한 답 (A)

> **“수학 수업(또는 자기주도 학습)에서 4×4 마방진 과제를 제출해야 하고, 행·열·대각선의 합이 모두 같아야 ‘완성’으로 인정받기 때문이다.”**

## 이 답에서 드러나는 불편함

| 불편 | 설명 |
|------|------|
| **완성 기준이 외부에 있다** | “맞다”의 기준이 본인의 이해가 아니라 채점·과제 규칙에 묶여 있음 |
| **중간 상태를 알기 어렵다** | 거의 다 맞았는데 한 줄만 어긋난 경우, 어디가 문제인지 찾는 데 시간이 많이 듦 |
| **수동 검증의 피로** | 16칸을 채운 뒤 행 4개·열 4개·대각 2개를 매번 더하는 과정이 반복적이고 실수하기 쉬움 |
| **시행착오 비용** | 숫자 배치를 바꿀 때마다 “이번엔 맞나?”를 처음부터 다시 확인해야 함 |
| **학습 목표와 수행 방식의 괴리** | 목표는 “규칙·대칭·패턴 이해”인데, 실제로 쓰는 시간 상당 부분이 **산술 검산**에 소모됨 |

## 구조적 문제

1. **검증 부담이 사고 부담보다 크다**  
   문제 해결(배치·추론)과 정답 확인(합 계산)이 분리되지 않아, 후자가 전자를 압도함.

2. **피드백이 늦고 거칠다**  
   “틀렸다”만 알 수 있고 “어느 제약이 깨졌는지”는 스스로 역추적해야 함.

3. **완성 ≠ 이해**  
   우연히 맞는 배치를 찾았거나, 예시를 베낀 경우 **왜 성립하는지**까지 검증되지 않음.

4. **확장성 없음**  
   종이·계산기 방식은 한 문제·한 번의 시도에 묶여, 같은 규칙을 다른 조건(일부 칸만 주어진 경우 등)에 재사용하기 어려움.

# STEP 3 — Why #2

## Q: 왜 단순 계산이 아니라 프로그램으로 구현하는가?

### 반복 가능성

- 같은 규칙을 **여러 번, 여러 변형**에 적용할 수 있음.
- “한 번 풀고 끝”이 아니라, 빈 칸 수·시작 조건·난이도가 바뀌어도 **동일한 판단 기준**을 유지할 수 있음.
- 학습자·개발자 모두 **재현 가능한 결과**를 기대할 수 있음.

### 검증 자동화

- 사람이 매번 하던 “합이 같은가?” 확인을 **일관된 절차**로 대체.
- 부분 완성 상태에서도 **지금까지 만족한 제약**과 **아직 깨진 제약**을 구분 가능.
- “거의 맞음”과 “완전히 틀림”의 차이를 **즉시·객관적으로** 전달 가능.

### 오류 방지

- 손계산·눈대중 검산에서 생기는 **누적 실수·누락**(한 줄만 안 본 경우 등)을 줄임.
- 중복 숫자, 범위 밖 숫자, 빈 칸 등 **규칙 위반을 조기에** 드러냄.
- “제출 직전에야 틀렸음을 발견”하는 상황을 줄이는 것이 목표.

### 규칙 기반 사고 훈련

- “감으로 맞춘다”에서 **“어떤 조건을 만족해야 하는가?”**로 사고를 옮김.
- 문제를 **입력(현재 격자 상태) → 판단(규칙 적용) → 결과(통과/위반)** 로 나누는 연습.
- 개발자로서 **요구를 명시 가능한 형태로 고정**하는 훈련이 됨.

### 이 단계에서의 한계 (솔직한 관찰)

프로그램은 **도구**일 뿐, “누가 왜 쓰는지”는 STEP 2만으로는 아직 답이 없음.  
**본인 학습용 도구**인지, **다른 학습자용**인지에 따라 “왜 프로그램인가”의 무게가 달라짐.

# STEP 4 — Why #3 (핵심 발견)

## Q: 왜 이 문제를 TDD 방식으로 설계하려 하는가?

### 무엇이 통제되어야 하는가?

| 통제 대상 | 이유 |
|-----------|------|
| **완성의 정의** | “다 채웠다”와 “규칙을 만족한다”를 혼동하지 않기 위해 |
| **부분 상태의 의미** | 일부만 채운 격자도 “지금 유효한가 / 아직 판단 불가인가”가 정해져야 함 |
| **판단 결과의 일관성** | 같은 격자 상태에 대해 항상 같은 pass/fail·위반 정보 |
| **변경의 영향 범위** | 한 칸을 바꿨을 때 **무엇만** 다시 확인하면 되는지 경계가 있어야 함 |
| **요구의 drift** | 구현이 진행되며 “대각선은 나중에”처럼 규칙이 흐려지는 것을 막기 위해 |

TDD는 여기서 “테스트를 먼저 쓴다”는 기술보다, **규칙을 먼저 고정하고 구현이 그 규칙을 따르게 한다**는 설계 태도에 가깝다.

### 어떤 불변 조건이 존재하는가?

4×4 마방진 맥락에서 **항상 참이어야 하는 성질** (알고리즘 없이, 규칙만):

1. **유일성** — 사용하는 숫자는 정해진 집합(예: 1~16)에서 각각 한 번만 등장.
2. **완전성** — “완성” 상태에서는 빈 칸이 없음.
3. **행 동일 합** — 모든 행의 합이 서로 같음.
4. **열 동일 합** — 모든 열의 합이 서로 같음.
5. **대각 동일 합** — 주대각·부대각의 합이 위와 같음 (문제 정의에 포함할 경우).
6. **매직 상수** — 4×4·1~16 설정에서는 “그 공통 합”이 **하나로 정해짐** (전체 합과 크기로부터).

**부분 완성**에서는:  
- (1) 위반 시 즉시 invalid  
- (2)(3)(4)(5)는 **아직 판단 불가** 또는 **이미 깨짐**으로 구분 가능해야 함.

### 왜 입력/출력이 명확해야 하는가?

| 이유 | 설명 |
|------|------|
| **판단 가능성** | “이 격자”가 들어왔을 때 무엇을 돌려줄지 애매하면 검증·피드백·학습 모두 불가 |
| **테스트 가능성** | 기대 결과를 문장/표로 적을 수 있어야 “맞게 만들었는지” 논쟁이 줄어듦 |
| **책임 분리** | “상태 해석”과 “규칙 적용”과 “사용자 메시지”가 섞이면 버그 위치를 찾기 어려움 |
| **점진적 구현** | 가장 단순한 입출력(예: 빈 격자 → invalid)부터 쌓을 수 있음 |
| **AI 협업** | Cursor와 작업할 때 **명시된 입출력 = 공유된 계약**; 모호하면 생성물이 매번 달라짐 |

**핵심 발견:**  
TDD를 택하는 진짜 이유는 “테스트 문화”가 아니라, **마방진이 본질적으로 ‘명시적 규칙 + 명시적 상태’ 문제**이기 때문이다. 규칙이 숫자로 쓸 수 있을 만큼 분명할수록, 설계·구현·학습이 같은 언어를 쓰게 된다.

---

# STEP 5 — 진짜 문제 정의

## 1. 표면 문제 정의 (잘못된 정의)

> **“4×4 마방진을 자동으로 만들고 검증하는 프로그램을 만든다.”**

**왜 잘못되었는가**

- **해결책(프로그램·자동 생성)** 이 문제 정의에 포함됨.
- **사용자·상황·불편**이 없음.
- **성공 기준**이 “동작한다” 수준으로만 끝남.
- **학습 목표(주니어 + AI + TDD)** 와 연결되지 않음.
- Mom Test 관점: 엄마에게 말해도 “그래서 누가 뭘 덜 힘들어해?”에 답이 없음.

---

## 2. 개선된 문제 정의 (정확한 정의)

> **“4×4 격자에 숫자를 배치할 때, ‘완성’이 무엇인지·‘지금 상태가 규칙을 어기는지’를 사람이 반복 계산하지 않고도 즉시·일관되게 알 수 있어야 한다.  
> 주니어 개발자는 그 판단 기준을 먼저 고정하고, 작은 단위로 검증 가능한 형태로 쌓아 올리며, AI 도구와 협업해 실제로 쓸 수 있는 최소한의 피드백 루프를 경험하려 한다.”**

**포함된 요소**

| 요소 | 내용 |
|------|------|
| **상황** | 격자 배치 + 완성/위반 판단 |
| **불편** | 반복 수동 검산, 늦은·거친 피드백 |
| **원하는 상태** | 즉시·일관·재현 가능한 판단 |
| **1차 사용자** | (가정) 같은 규칙으로 학습·연습하는 본인 또는 동료 학습자 |
| **2차 목표** | 규칙을 명시하고 검증 가능하게 쌓는 **개발 사고 훈련** |

---

## 3. 이 문제의 핵심 Invariant

**한 문장:**  
> **주어진 4×4 배치에서, “완성”으로 인정되려면 정해진 숫자 집합이 중복·누락 없이 한 번씩 쓰이고, 모든 행·열·(정의된) 대각선의 합이 동일해야 한다.**

**설계·학습에서의 역할**

- 구현이 어떤 기능을 추가하든 **이 성질을 깨뜨리면 안 됨**.
- 부분 상태에서는: **위반은 즉시 드러나고**, **아직 말할 수 없는 것은 “미정”** 으로 남김.
- UI·힌트·난이도 등은 모두 이 invariant **위**에 올라가는 레이어.

---

## 4. 우리가 실제로 훈련하려는 사고 능력

| 능력 | 훈련 내용 |
|------|-----------|
| **문제 vs 해결책 분리** | “무엇이 참이어야 하는가”를 먼저 쓰고, 도구 선택은 나중 |
| **불변 조건 식별** | 변하는 것(격자 내용)과 변하지 않아야 할 것(규칙) 구분 |
| **계약적 사고** | 입력(상태)과 출력(판단)을 말로 합의 가능하게 고정 |
| **점진적 신뢰 쌓기** | 가장 작은 참인 경우부터 “맞다”를 쌓아 올림 (TDD 마인드) |
| **피드백 루프 설계** | 사용자(또는 본인)가 **다음 행동**을 알 수 있는 정보 설계 |
| **AI 협업** | 모호한 요구 대신 **검증 가능한 진술**을 AI와 공유하는 습관 |
| **범위 통제** | 4×4·고정 규칙 등 **의도적 제한**으로 학습과 품질 동시 확보 |

# Magic Square 프로젝트 — 문제 정의 보고서

**작성 목적:** 4×4 Magic Square 프로그램 개발 전, 문제 인식 및 정의 단계 정리  
**대상:** 주니어 개발자 + Cursor AI 활용 학습 프로젝트  
**범위:** 관찰 → Why 분석 → 진짜 문제 정의 (구현·설계·코드·알고리즘 제외)  
**작성일:** 2026-05-28

---

## Executive Summary

본 프로젝트의 표면 목표는 “4×4 마방진 프로그램”이지만, 문제 정의 단계를 거치며 **실제 다루는 문제**는 다음과 같이 정리된다.

> **4×4 격자에 숫자를 배치할 때, ‘완성’이 무엇인지·‘현재 상태가 규칙을 어기는지’를 사람이 반복 계산하지 않고도 즉시·일관되게 알 수 있어야 한다. 주니어 개발자는 그 판단 기준을 먼저 고정하고, 작은 단위로 검증 가능한 형태로 쌓아 올리며, AI 도구와 협업해 실제로 쓸 수 있는 최소한의 피드백 루프를 경험하려 한다.**

---

## STEP 1 — Observation (관찰)

### 1.1 현재 해결하려는 상황

- 주니어 개발자가 **실제로 동작하는 소프트웨어**를 만들며 역량을 쌓고자 함.
- **Cursor AI**를 코드 생성기가 아닌 **협업 도구**로 활용하는 법을 익히려는 시도.
- 수학·퍼즐·알고리즘 영역은 **정답이 명확**해 학습·검증에 유리함.
- 워크스페이스(`MagicSquare_`)는 아직 비어 있으며, **구체적 사용자·상황·불편**은 드러나지 않음.

**한 줄 요약:** “개발 실력 + AI 활용 + 검증 가능한 도메인”을 동시에 만족시키려는 **학습·실험 상황**이며, **누구의 어떤 문제인지는 아직 관찰되지 않음.**

### 1.2 왜 4×4 마방진 유형의 문제인가

| 관찰 | 의미 |
|------|------|
| 규모가 작음 (4×4, 1~16) | 상태 공간·UI·테스트 범위가 다루기 쉬움 |
| 규칙이 명확함 | 맞다/틀리다를 자동 검증하기 좋음 |
| 생성·검증 분리 가능 | 설계·테스트 연습에 적합 |
| 시각적으로 직관적 | 격자 UI, 합 표시 등 피드백 단서 풍부 |
| 교육·퍼즐 맥락과 연결 | 잠재 사용자(학생, 교사, 퍼즐 애호가) 상상 용이 |

**주의:** “4×4가 적당해서”는 **개발자 편의**에 가깝다. 사용자가 4×4를 꼭 필요로 하는지는 아직 검증되지 않음.

### 1.3 학습·시스템 설계 맥락

**학습 맥락**

- 초·중등 수학: 대칭, 합, 패턴, 논리적 추론
- 알고리즘·자료구조: 제약 만족, 불변 조건 검증
- 소프트웨어 공학: 입력 검증, 상태 표현, 단위·통합 테스트
- AI 협업: 문제 정의 → 스펙 → 구현 → 리팩터 프로세스 학습

**시스템 설계 맥락 (향후 자연스럽게 등장할 질문)**

- 생성 vs 검증 vs 힌트 vs 풀이 — 역할 분리
- 상태 모델 — 무엇이 “진실의 원천”인가
- 피드백 루프 — 틀렸을 때 어디까지 설명할 것인가
- 난이도·모드 — 빈 칸 채우기, 단계별 힌트 등
- 확장성 — 4×4 고정 vs n×n 일반화

### 1.4 STEP 1 관찰 요약

| 관찰됨 | 아직 관찰되지 않음 |
|--------|-------------------|
| 주니어 + AI + 작고 검증 가능한 프로젝트 의도 | 특정 사용자의 구체적 불편 |
| 작은 격자 + 명확한 규칙 + 시각화 가능 | 4×4가 사용자에게 필수인 이유 |
| 교육·퍼즐·알고리즘과 연결 가능 | 실제 사용 맥락(교실, 혼자 공부, 면접 등) |

---

## STEP 2 — Why #1

### 2.1 가정한 답 (A)

> **“수학 수업(또는 자기주도 학습)에서 4×4 마방진 과제를 제출해야 하고, 행·열·대각선의 합이 모두 같아야 ‘완성’으로 인정받기 때문이다.”**

### 2.2 드러나는 불편함

| 불편 | 설명 |
|------|------|
| 완성 기준이 외부에 있음 | “맞다”의 기준이 본인 이해가 아닌 채점·과제 규칙에 묶임 |
| 중간 상태 파악 어려움 | 한 줄만 어긋난 경우 원인 찾기에 시간 소모 |
| 수동 검증 피로 | 16칸 후 행·열·대각 합을 반복 계산, 실수하기 쉬움 |
| 시행착오 비용 | 배치 변경마다 처음부터 재검증 |
| 학습 목표와 수행 방식의 괴리 | “규칙 이해”보다 “산술 검산”에 시간 소모 |

### 2.3 구조적 문제

1. **검증 부담 > 사고 부담** — 배치·추론과 정답 확인이 분리되지 않음  
2. **피드백이 늦고 거침** — “틀렸다”만 알고, 어느 제약이 깨졌는지 역추적 필요  
3. **완성 ≠ 이해** — 우연히 맞거나 예시를 베낀 경우, 왜 성립하는지 검증 안 됨  
4. **확장성 없음** — 종이·계산기 방식은 한 문제·한 시도에 묶임  

---

## STEP 3 — Why #2

### 3.1 Q: 왜 단순 계산이 아니라 프로그램으로 구현하는가?

#### 반복 가능성

- 같은 규칙을 여러 번, 여러 변형에 적용 가능  
- 빈 칸 수·시작 조건·난이도가 바뀌어도 동일한 판단 기준 유지  
- 학습자·개발자 모두 재현 가능한 결과 기대 가능  

#### 검증 자동화

- “합이 같은가?” 확인을 일관된 절차로 대체  
- 부분 완성 상태에서 만족/위반 제약 구분  
- “거의 맞음”과 “완전히 틀림”을 즉시·객관적으로 전달  

#### 오류 방지

- 손계산·눈대중 검산의 누적 실수·누락 감소  
- 중복 숫자, 범위 밖 숫자, 빈 칸 등 조기 발견  
- “제출 직전에야 틀림 발견” 상황 감소  

#### 규칙 기반 사고 훈련

- “감으로 맞추기” → “어떤 조건을 만족해야 하는가?”  
- 입력(격자 상태) → 판단(규칙 적용) → 결과(통과/위반) 분리 연습  
- 요구를 명시 가능한 형태로 고정하는 개발자 습관 형성  

### 3.2 한계

프로그램은 **도구**일 뿐. 본인 학습용인지, 다른 학습자용인지에 따라 “왜 프로그램인가”의 무게가 달라짐.

---

## STEP 4 — Why #3 (핵심 발견)

### 4.1 Q: 왜 TDD 방식으로 설계하려 하는가?

#### 통제되어야 하는 것

| 통제 대상 | 이유 |
|-----------|------|
| 완성의 정의 | “다 채웠다” ≠ “규칙 만족” 혼동 방지 |
| 부분 상태의 의미 | 유효 / 판단 불가 / 위반 구분 |
| 판단 결과의 일관성 | 같은 상태 → 같은 pass/fail |
| 변경의 영향 범위 | 한 칸 변경 시 재확인 범위 명확화 |
| 요구의 drift | 구현 중 규칙 흐려짐 방지 |

TDD는 “테스트를 먼저 쓴다”기보다 **규칙을 먼저 고정하고 구현이 따르게 한다**는 설계 태도.

#### 존재하는 불변 조건 (Invariant)

1. **유일성** — 정해진 숫자 집합(1~16)에서 각각 한 번만 등장  
2. **완전성** — “완성” 상태에서 빈 칸 없음  
3. **행 동일 합** — 모든 행의 합이 서로 같음  
4. **열 동일 합** — 모든 열의 합이 서로 같음  
5. **대각 동일 합** — 주대각·부대각 합이 위와 같음 (정의에 포함 시)  
6. **매직 상수** — 4×4·1~16 설정에서 공통 합이 하나로 정해짐  

**부분 완성:** (1) 위반 시 즉시 invalid; (2)~(5)는 “미정” 또는 “이미 깨짐”으로 구분.

#### 입력/출력이 명확해야 하는 이유

| 이유 | 설명 |
|------|------|
| 판단 가능성 | 애매한 입출력 → 검증·피드백 불가 |
| 테스트 가능성 | 기대 결과를 문장/표로 적을 수 있어야 함 |
| 책임 분리 | 상태 해석 / 규칙 적용 / 사용자 메시지 분리 |
| 점진적 구현 | 가장 단순한 경우부터 쌓기 가능 |
| AI 협업 | 명시된 입출력 = 공유 계약 |

### 4.2 핵심 발견

TDD를 택하는 이유는 “테스트 문화”가 아니라, **마방진이 본질적으로 ‘명시적 규칙 + 명시적 상태’ 문제**이기 때문이다.

---

## STEP 5 — 진짜 문제 정의

### 5.1 표면 문제 정의 (잘못된 정의)

> **“4×4 마방진을 자동으로 만들고 검증하는 프로그램을 만든다.”**

**문제점**

- 해결책(프로그램·자동 생성)이 정의에 포함됨  
- 사용자·상황·불편 없음  
- 성공 기준이 “동작한다” 수준  
- 학습 목표와 미연결  
- Mom Test: “누가 뭘 덜 힘들어해?”에 답 없음  

### 5.2 개선된 문제 정의 (정확한 정의)

> **“4×4 격자에 숫자를 배치할 때, ‘완성’이 무엇인지·‘지금 상태가 규칙을 어기는지’를 사람이 반복 계산하지 않고도 즉시·일관되게 알 수 있어야 한다.  
> 주니어 개발자는 그 판단 기준을 먼저 고정하고, 작은 단위로 검증 가능한 형태로 쌓아 올리며, AI 도구와 협업해 실제로 쓸 수 있는 최소한의 피드백 루프를 경험하려 한다.”**

| 요소 | 내용 |
|------|------|
| 상황 | 격자 배치 + 완성/위반 판단 |
| 불편 | 반복 수동 검산, 늦고 거친 피드백 |
| 원하는 상태 | 즉시·일관·재현 가능한 판단 |
| 1차 사용자 | (가정) 같은 규칙으로 학습·연습하는 본인 또는 동료 학습자 |
| 2차 목표 | 규칙을 명시하고 검증 가능하게 쌓는 개발 사고 훈련 |

### 5.3 핵심 Invariant

> **주어진 4×4 배치에서, “완성”으로 인정되려면 정해진 숫자 집합이 중복·누락 없이 한 번씩 쓰이고, 모든 행·열·(정의된) 대각선의 합이 동일해야 한다.**

- 구현 기능 추가 시 **이 성질을 깨뜨리면 안 됨**  
- 부분 상태: 위반은 즉시 드러내고, 아직 말할 수 없는 것은 “미정”  
- UI·힌트·난이도는 invariant **위** 레이어  

### 5.4 훈련하려는 사고 능력

| 능력 | 훈련 내용 |
|------|-----------|
| 문제 vs 해결책 분리 | “무엇이 참이어야 하는가”를 먼저 |
| 불변 조건 식별 | 변하는 것 vs 변하지 않아야 할 것 |
| 계약적 사고 | 입출력을 말로 합의 가능하게 고정 |
| 점진적 신뢰 쌓기 | 가장 작은 참인 경우부터 |
| 피드백 루프 설계 | 다음 행동을 알 수 있는 정보 |
| AI 협업 | 검증 가능한 진술을 AI와 공유 |
| 범위 통제 | 4×4·고정 규칙 등 의도적 제한 |
